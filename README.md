# Портфолио системного аналитика: Проект «Zoo Ticket»

👋 Привет! Меня зовут Татьяна Дементьева. Это портфолио учебного проекта по покупке билетов в зоопарк. Здесь я показываю цикл работы: от модели данных до прототипа интерфейса.

**Инструменты и нотации:** BPMN, UML (Sequence, Activity, State), ERD, SQL, Figma, OpenAPI (Swagger), Draw.io.

---

## 📦 Артефакты проекта

### 1. Модель данных системы (ER-диаграмма)
**Задача:** Спроектировать логическую схему базы данных для билетной системы зоопарка.  
**Решение:** Разработана ER-диаграмма в нотации Crow's Foot с выделением сущностей: `Посетитель`, `Билет`, `Турникет`, `Платеж`. Проведена нормализация до 3НФ.  
📎 [Посмотреть диаграмму](./1-ER-diagram/er_diagram.png)

---

### 2. Логика работы оборудования (Sequence-диаграмма)
**Задача:** Описать взаимодействие пользователя, турникета и сервера при входе в зоопарк.  
**Решение:** Построена Sequence-диаграмма, которая показывает, как система проверяет валидность QR-кода и открывает турникет.  
📎 [Посмотреть диаграмму](./2-sequence-diagram/sequence_diagram.png)

---

### 3. Прототип интерфейса (Figma)
**Задача:** Разработать удобный интерфейс для покупки билета.  
**Решение:** Создан прототип в Figma, включающий выбор даты, количества билетов и оплату.  
📎 [Посмотреть скриншот](./3-figma-prototype/payment_prototype.png)

---

### 4. Спецификация API платежной системы
**Задача:** Описать контракт для интеграции с внешней платежной системой.  
**Решение:** Составлена документация API с описанием эндпоинтов для создания и подтверждения платежа.  
📎 [Открыть спецификацию API](./4-api-documentation/ZooTicketPaymentAPI.docx)

---

### 5. Бизнес-процесс возврата билета (Activity-диаграмма)
**Задача:** Визуализировать алгоритм возврата денежных средств и аннулирования билета.  
**Решение:** Создана Activity-диаграмма с ветвлениями: проверка срока возврата, расчет суммы, изменение статуса билета.  
📎 [Посмотреть диаграмму](./5-activity-diagram/activity_diagram.png)

---

### 6. Жизненный цикл билета (State Machine Diagram)
**Задача:** Описать все возможные состояния билета и правила перехода между ними.  
**Решение:** Создана диаграмма состояний, показывающая путь билета от «Создан» до «Использован» или «Возвращен».  
📎 [Посмотреть диаграмму состояний](./6-state-diagram/state_mashine_diagram.png)

---

### 7. Спецификация требований (SRS)
**Задача:** Собрать и структурировать бизнес-требования к билетной системе.  
**Решение:** Составлен документ с функциональными и нефункциональными требованиями, а также Use Cases.  
📎 [Открыть требования](./7-requirements/requirements.docx)
